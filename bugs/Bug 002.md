# Bug Fix Specification – Stabilize Backend Before Chapters Frontend Migration

## Objective

Restore the application to a stable state where:

* Existing functionality works without regression.
* Journal creation works.
* Journal loading works.
* Entry creation works.
* Entry retrieval works.
* Journal page loads correctly.
* Authentication continues working.
* The backend becomes a stable foundation for the upcoming Chapters frontend redesign.

No frontend redesign should be implemented as part of this task.

---

# Background

The project is currently transitioning from **Good Time Journal** to **Chapters**.

During this migration, several backend APIs and database changes were partially introduced, resulting in inconsistent routing and broken entry creation.

The immediate goal is **not** to implement the new architecture.

The goal is to restore a stable backend first.

---

# Current Problems

## 1. Mixed API Architecture

The project currently contains two API designs.

Legacy

```
/api/journal
```

New

```
/api/entries
```

Both are partially implemented.

Result:

* Existing pages call one API.
* New code calls another API.
* Routes are inconsistent.

---

## 2. Entry Summary API introduced too early

A new endpoint

```
GET /api/entries
```

was added before completing the migration.

This introduced routing conflicts and changed controller exports.

---

## 3. Controller Export Inconsistency

The controller originally used

```js
module.exports = {
    ...
}
```

A new

```js
exports.getEntrySummaries
```

was later appended.

This caused

```
TypeError:
argument handler must be a function
```

This has been corrected but should be verified.

---

## 4. Database Recreation

The SQLite database was deleted and recreated.

Schema now differs from the original migration logic.

Need to verify:

* users
* journals
* journal_entries

are created correctly.

---

## 5. Save Entry Failure

Current symptom:

Browser

```
Unable to save journal entry.
```

Need to identify the actual backend cause.

Possible causes:

* incorrect POST endpoint
* journal lookup failure
* validation failure
* SQL insert failure

Need proper logging.

---

## 6. Journal Loading Issues

Some pages display

```
Journal could not be found.
```

Need to verify:

```
GET /api/journals/:id
```

works correctly.

---

# Required Debugging

Replace generic backend errors like

```
Unable to save journal entry.
```

with actual logged SQL errors.

Example

Current

```js
return res.status(500).json({
    error: "Unable to save journal entry."
});
```

Temporarily change to

```js
console.error(err);

return res.status(500).json({
    error: err.message
});
```

until backend is stable.
